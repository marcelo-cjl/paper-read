## 背景与动机

### 要解决什么问题？

LLM 模型参数量巨大，需要高效压缩。标量量化（每个参数独立量化）已触及信息论的极限：

```
┌─────────────────────────────────────────────────────────────┐
│             标量 vs 向量量化                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  标量量化 (每个参数独立处理):                                │
│  ┌─────────────────────────────────────┐                   │
│  │  w₁ → q₁    w₂ → q₂    w₃ → q₃   │                   │
│  │  ↓          ↓          ↓           │                   │
│  │  独立量化   独立量化   独立量化     │                   │
│  │                                     │                   │
│  │  受 Shannon 率失真理论下界限制      │                   │
│  │  无法进一步改善!                    │                   │
│  └─────────────────────────────────────┘                   │
│                                                             │
│  向量量化 (参数块联合编码):                                  │
│  ┌─────────────────────────────────────┐                   │
│  │  [w₁, w₂, ..., w₂₄] → q           │                   │
│  │       ↓                             │                   │
│  │  联合量化 24 个参数                 │                   │
│  │                                     │                   │
│  │  可以突破标量量化的理论极限!        │                   │
│  │  利用参数间的统计相关性             │                   │
│  └─────────────────────────────────────┘                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 向量量化的挑战

```
┌─────────────────────────────────────────────────────────────┐
│            传统向量量化的实际困难                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 码本存储开销                                            │
│     ┌────────────────────────────────────┐                 │
│     │  需要存储巨大的查找表              │                 │
│     │  码本本身占用大量内存              │                 │
│     └────────────────────────────────────┘                 │
│                                                             │
│  2. 查找复杂度                                              │
│     ┌────────────────────────────────────┐                 │
│     │  在大码本中找最近码字              │                 │
│     │  时间复杂度随码本大小线性增长      │                 │
│     └────────────────────────────────────┘                 │
│                                                             │
│  3. 反量化效率                                              │
│     ┌────────────────────────────────────┐                 │
│     │  推理时需要快速解码                │                 │
│     │  查表操作难以并行化                │                 │
│     └────────────────────────────────────┘                 │
│                                                             │
│  LLVQ 的解决方案: 使用数学结构化的 Leech 格子               │
│  → 无需存储码本、有高效搜索算法、可完全并行化              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 数学背景: 什么是 Leech 格子？

### 直觉理解

```
┌─────────────────────────────────────────────────────────────┐
│              Leech 格子的直觉理解                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  想象在空间中堆放球体，使得:                                │
│  1. 球体之间没有间隙 (密堆积)                               │
│  2. 排列具有规律性 (格子结构)                               │
│                                                             │
│  2D: 蜂巢排列 (最优)                                       │
│       ○ ○ ○ ○                                              │
│      ○ ○ ○ ○                                               │
│       ○ ○ ○ ○                                              │
│                                                             │
│  3D: FCC/HCP 排列 (开普勒猜想)                             │
│                                                             │
│  24D: Leech 格子 (已证明最优!)                              │
│                                                             │
│  ┌────────────────────────────────────────┐                │
│  │  Leech 格子是 24 维空间中的                │                │
│  │  最密球堆积，也是唯一的最优格子        │                │
│  │                                        │                │
│  │  每个点有 196560 个"接吻邻居"         │                │
│  │  (24维空间中可同时接触的球数)          │                │
│  └────────────────────────────────────────┘                │
│                                                             │
│  量化意义: 格子点就是码字                                   │
│  最密堆积 = 最小量化误差!                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Leech 格子与 Extended Golay Code

```
┌─────────────────────────────────────────────────────────────┐
│            Leech 格子的构造基础                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Extended Golay Code (扩展Golay码):                         │
│  ┌─────────────────────────────────────┐                   │
│  │  [24, 12, 8] 二进制码               │                   │
│  │  24位码字，12位信息位               │                   │
│  │  最小汉明距离 = 8                   │                   │
│  │                                     │                   │
│  │  拥有 4096 个码字                   │                   │
│  └─────────────────────────────────────┘                   │
│                                                             │
│  Leech 格子通过 Golay 码构造:                               │
│  ┌─────────────────────────────────────┐                   │
│  │  格子点 = Golay码字的组合变换       │                   │
│  │                                     │                   │
│  │  关键优势:                          │                   │
│  │  • 结构已知 → 无需存储码本          │                   │
│  │  • 有代数结构 → 高效搜索算法        │                   │
│  │  • 可用比特串编码 → 支持索引转换    │                   │
│  └─────────────────────────────────────┘                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Phase 2: Leech 格子量化

### 2.1 目的

将每个 24 维权重块映射到最近的 Leech 格子点，并编码为紧凑的比特串。

### 2.2 基于 Golay 码的搜索算法

```
┌─────────────────────────────────────────────────────────────┐
│               Leech 格子最近点搜索                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  输入: 24 维权重块 w = [w₁, w₂, ..., w₂₄]                 │
│  输出: 最近的 Leech 格子点 λ 及其索引                       │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                     │   │
│  │  Step 1: 投影到 Golay 码的陪集                      │   │
│  │     w → 多个候选陪集                                │   │
│  │                                                     │   │
│  │  Step 2: 在每个陪集中找最近格子点                   │   │
│  │     利用 Golay 码的代数结构                         │   │
│  │     快速解码算法                                     │   │
│  │                                                     │   │
│  │  Step 3: 比较候选点，选择最近的                     │   │
│  │     λ* = argmin ‖w - λ‖²                           │   │
│  │            λ∈候选集                                  │   │
│  │                                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  关键: 不需要遍历所有格子点!                                │
│  利用代数结构，搜索复杂度与码本大小无关                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 LLVQ 的三项改进

```
┌─────────────────────────────────────────────────────────────┐
│              LLVQ 的三项关键改进                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  改进 1: 无码本索引机制                                     │
│  ┌─────────────────────────────────────────────────┐       │
│  │                                                 │       │
│  │  传统 VQ:   码字 → 查表 → 索引号               │       │
│  │             索引号 → 查表 → 码字               │       │
│  │             需要存储码本!                        │       │
│  │                                                 │       │
│  │  LLVQ:      格子点 ◀══▶ 比特串                 │       │
│  │             通过数学公式直接转换!               │       │
│  │             无需存储任何码本!                    │       │
│  │                                                 │       │
│  └─────────────────────────────────────────────────┘       │
│                                                             │
│  改进 2: 跨壳层角度搜索                                     │
│  ┌─────────────────────────────────────────────────┐       │
│  │                                                 │       │
│  │  Leech 格子有不同的"壳层" (不同半径)            │       │
│  │                                                 │       │
│  │        ╱ ─ ─ ╲     壳层 3                      │       │
│  │      ╱  ╱─╲   ╲    壳层 2                      │       │
│  │     │  │ ● │   │   壳层 1                      │       │
│  │      ╲  ╲─╱   ╱    原点                        │       │
│  │        ╲ ─ ─ ╱                                  │       │
│  │                                                 │       │
│  │  LLVQ 支持跨壳层搜索最优量化点                  │       │
│  │  → 更灵活的量化精度控制                         │       │
│  │                                                 │       │
│  └─────────────────────────────────────────────────┘       │
│                                                             │
│  改进 3: 完全并行化反量化内核                               │
│  ┌─────────────────────────────────────────────────┐       │
│  │                                                 │       │
│  │  推理时: 比特串 → 格子点 → 权重值               │       │
│  │                                                 │       │
│  │  每个块的反量化相互独立                         │       │
│  │  → GPU 上完全并行执行                           │       │
│  │  → 推理速度不受量化方法拖累                     │       │
│  │                                                 │       │
│  └─────────────────────────────────────────────────┘       │
```

## 总结

LLVQ 将数学中的 Leech 格子引入 LLM 压缩，实现了理论与实践的完美结合：

1. **最优堆积**: 24 维 Leech 格子提供最密球堆积，意味着最小量化误差
2. **无码本**: 格子结构由数学定义，无需显式存储码本
3. **高效搜索**: 基于 Extended Golay Code 的代数搜索算法
4. **完全并行**: 反量化内核可在 GPU 上完全并行执行

核心突破：用数学结构替代暴力码本，同时获得了更好的量化性能和更低的存储/计算开销。
